# Architectural Review: lucos_docker_health

**Date:** 2026-04-05
**Reviewer:** lucos-architect[bot]
**Commit:** 023a133

## Summary

A clean, well-scoped service that does exactly what it should and nothing more. The push-model design (ADR-0001) was the right call — it eliminated per-host infrastructure requirements and works on all three hosts with the same image. The implementation is a ~180-line Go binary with no unnecessary abstractions. The main gaps are in security hygiene tooling (Dependabot coverage, CodeQL), not in the code itself.

## Strengths

- **Tight scope.** The binary does one thing: read Docker healthchecks, POST to schedule_tracker. No HTTP server, no state, no configuration complexity. This is exactly the right level of complexity for the problem.
- **ADR is honest about trade-offs.** The security section acknowledges that running as root was a pragmatic concession, not a design goal. The negative consequences section is substantive — it names the schedule_tracker dependency risk and the 60-second staleness window. Too many ADRs only list positives.
- **Heartbeat healthcheck pattern.** The `--healthcheck` flag reusing the same binary to check a heartbeat file is elegant. It works on distroless (exec form, no shell), actually verifies the main loop is running (not just that an HTTP server responds), and satisfies the Docker convention. Good pattern for other push-only services.
- **Correct separation of monitoring layers.** The Docker healthcheck measures "is the process alive", schedule_tracker's stale-check measures "are reports arriving". These are complementary, not redundant. A deadlocked process fails the heartbeat; a network partition fails the stale-check. Neither alone covers both failure modes.
- **CLAUDE.md is practical.** It includes a local testing workflow with a concrete `docker run` command. It's honest about the lack of tests and explains why ("coverage is provided by CI build verification and production monitoring"). It's the right length for the repo — no cruft.
- **Environment variable handling follows conventions.** `SCHEDULE_TRACKER_ENDPOINT` (not a URL constructor), `SYSTEM` from lucos_creds, `HOSTDOMAIN` for per-host differentiation. The `User-Agent` header on the schedule_tracker POST is set correctly per ADR-0001 in the lucos repo.

## Concerns

- **Dependabot does not monitor Go modules.** The `.github/dependabot.yml` covers `github-actions` and `docker` but not `gomod`. The Docker SDK (`github.com/docker/docker`) and its ~20 transitive dependencies (including OpenTelemetry and containerd packages) will not receive automated security update PRs. For a service with Docker socket access, this is the most important gap. (#43)
- **No CodeQL static analysis.** Other lucos repos have CodeQL configured. This service's Docker socket access makes it security-sensitive — static analysis would catch issues in the code or patterns that could be exploitable. (#45)
- **Compiled binary not in `.gitignore`.** Running `go build` locally produces a ~10MB `lucos_docker_health` binary that shows as untracked. Minor, but easy to fix. (#44)
- **High issue churn during implementation.** The repo has 38 closed issues for what is essentially a 180-line binary, a Dockerfile, and a compose file. Notable: the Docker socket permissions required three rounds of fixes (#33/#34, #37, #38) and the healthcheck CMD prefix needed a follow-up (#35/#36). This suggests implementations were pushed to production without sufficient local testing — despite the CLAUDE.md documenting a local test procedure. This is a process concern, not an architectural one, but worth noting as a pattern to watch across agent-implemented services.
